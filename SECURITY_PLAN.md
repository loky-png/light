# План улучшения безопасности Light Messenger

## Текущие проблемы:
1. ❌ HTTP вместо HTTPS - токены и пароли передаются открытым текстом
2. ❌ Сообщения не шифруются - сервер видит весь контент
3. ❌ Много мелких запросов - неэффективно и медленно
4. ✅ Пароли хешируются bcrypt на сервере (уже есть)

## Решения:

### 1. HTTPS (КРИТИЧНО)
- Настроить SSL сертификат на nginx
- Использовать Let's Encrypt для бесплатного SSL
- Изменить все URL с `http://` на `https://`
- WebSocket тоже через WSS (защищенный)

### 2. End-to-End шифрование сообщений
- Генерация ключевой пары (публичный/приватный) для каждого пользователя
- Публичный ключ хранится на сервере
- Приватный ключ хранится ТОЛЬКО на клиенте (localStorage зашифрованный)
- Сообщения шифруются публичным ключом получателя
- Расшифровать может только получатель своим приватным ключом
- Сервер видит только зашифрованный текст

### 3. Оптимизация запросов (пакетирование)
- Новый endpoint `/api/sync` - возвращает все данные одним запросом:
  - Список чатов
  - Статусы пользователей
  - Счетчики непрочитанных
  - Последние сообщения
- Уменьшение количества запросов с 5-10 до 1-2

### 4. Дополнительная защита
- Rate limiting - ограничение количества запросов
- JWT токены с коротким временем жизни + refresh токены
- Защита от CSRF атак
- Валидация всех входных данных

## Приоритет реализации:
1. **HTTPS** (самое важное) - без этого все остальное бесполезно
2. **Пакетирование запросов** - улучшит производительность
3. **E2E шифрование** - максимальная приватность
4. **Дополнительная защита** - финальная полировка

## Что делать сейчас:
Начнем с HTTPS, потом пакетирование, потом E2E шифрование.
